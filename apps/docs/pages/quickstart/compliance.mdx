# Quickstart: Compliance mapping

Every Pencheff scan &mdash; URL, Repo, or LLM &mdash; carries a
compliance rollup that fans every active finding out across the
frameworks that match the target&rsquo;s asset class. The rollup is
deterministic, recomputed on read, and consumable from the web UI,
the JSON / CSV / DOCX / Markdown exports, and the public REST API.

## 1. Pick the target kind

| Target kind | Frameworks emitted |
| --- | --- |
| URL (DAST) | OWASP Top 10 (2021) &middot; PCI-DSS 4.0 &middot; NIST 800-53 Rev 5 &middot; SOC 2 &middot; ISO 27001:2022 &middot; HIPAA |
| Repo (SAST · SCA · IaC · secrets) | Same six. RepoFinding rows infer a category from scanner + rule_id. |
| LLM (red team) | OWASP LLM Top 10 (2025) &middot; MITRE ATLAS &middot; NIST AI Risk Management Framework &middot; EU AI Act |

The framework set is fixed per target kind &mdash; the API does not
let a caller request HIPAA controls for an LLM endpoint, since none
of the LLM finding categories map onto HIPAA controls.

## 2. Open the rollup

<Tabs items={["URL / LLM scan", "Repo scan", "API"]}>
<Tabs.Tab>

1. Open the assessment page at `/scans/{id}`.
2. Scroll to the **§ Compliance mapping** card.
3. Click **View compliance mapping &rarr;**.

The page mirrors the layout of `/scans/{id}/threat-model`:

* A summary strip (`Findings`, `Controls touched`, `Frameworks`,
  `Target`).
* A horizontal framework picker that switches between the six (or
  four, for LLM) frameworks without a page reload.
* A control table with finding counts and a severity ribbon
  (C / H / M / L / I).
* A per-finding mapping table for auditors who want to see the
  forward direction (`finding &rarr; controls`) instead of the reverse
  (`control &rarr; findings`).

</Tabs.Tab>
<Tabs.Tab>

1. Open the repo scan at `/repos/scans/{id}`.
2. Click **View compliance mapping &rarr;** in the action row.

The repo compliance page reads from the same `build_compliance_rollup`
service and renders the identical wire shape &mdash; only the source
table changes (`repo_findings` instead of `findings`).

</Tabs.Tab>
<Tabs.Tab>

```bash
# URL or LLM scan
curl -H "Authorization: Bearer $PENCHEFF_API_KEY" \
  "$PENCHEFF_API_BASE/scans/$SCAN_ID/compliance" | jq

# Repo scan
curl -H "Authorization: Bearer $PENCHEFF_API_KEY" \
  "$PENCHEFF_API_BASE/repos/scans/$SCAN_ID/compliance" | jq
```

Sample slice of the response (full shape on the [feature

## 3. Ship it as audit evidence

* **DOCX / Markdown report** &mdash; the compliance rollup is rendered
  into a `Compliance` appendix between the executive summary and the
  findings register, listing every control that fired with its
  severity breakdown.
* **JSON / CSV exports** &mdash; every finding carries a `compliance`
  key (`{ "OWASP Top 10": ["A03: Injection"], "PCI-DSS": ["6.5.1"], … }`)
  so a downstream pipeline can ingest mappings without re-running the
  categoriser.
* **SBOM attachment** &mdash; for SOC 2 / ISO 27001 / NIST 800-53
  attestations, attach the [generated SBOM](/quickstart/sbom) as
  source-of-record evidence for SR-3 / SC-12 / A.5.21 controls.

<Callout type="info">
  **Same shape, three target kinds.** The two endpoints
  (`/scans/{id}/compliance` and `/repos/scans/{id}/compliance`) return
  the **identical** JSON envelope so the same web component, JSON
  exporter, and report appendix can consume any scan id without
  branching by target kind.
</Callout>
