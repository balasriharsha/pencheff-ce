# Contributing to Pencheff

Thanks for considering a contribution. This file covers the **mechanical**
prerequisites for getting a PR merged. The architectural and stylistic
conventions live alongside the code (per-package `README.md`, the docs
site at `apps/docs/`, and the `AGENTS.md` at the repo root).

## Developer Certificate of Origin (DCO)

Every commit must carry a `Signed-off-by:` trailer. By signing off,
you certify the [Developer Certificate of Origin](https://developercertificate.org/)
— that you own (or have permission to contribute) the code, and that
it can be released under Pencheff's MIT license.

Use `git commit -s` (or `--signoff`) on every commit. The DCO bot at
`.github/workflows/dco.yml` blocks PRs that contain unsigned commits.

```
Signed-off-by: Random J Developer <random@developer.example.org>
```

## License compatibility

Pencheff ships under the MIT license. Inputs we depend on must be
**MIT, Apache 2.0, BSD-2 / BSD-3, ISC, CC0, or U.S. public domain**.
CC-BY is acceptable when the attribution is preservable in our output.

* **No GPL / AGPL / SSPL / BSL imports** — full stop.
* **LGPL is subprocess-only** — never linked. If you wrap an LGPL
  binary, do it through `plugins/pencheff/pencheff/core/tool_runner.py`
  (which already runs as a subprocess).
* **Do not import** from any commercial product's data feeds, rule
  packs, or proprietary APIs without an explicit license. Specifically
  forbidden as inputs: Snyk Vulnerability DB, Burp Suite bChecks,
  Checkmarx rules, Veracode signatures, Semgrep Pro Engine / Pro rules.

The license-audit CI at `.github/workflows/license-audit.yml` runs on
every PR and will fail if a new dependency lands without an entry in
`tools/license-allowlist.txt` or if its declared license does not pass
the allowlist policy.

## SPDX headers

New source files (`.py`, `.sh`, `.js`, `.jsx`, `.ts`, `.tsx`, `.cjs`,
`.mjs`, `.go`, `.rs`) must declare:

```
SPDX-License-Identifier: MIT
```

near the top of the file. `tools/spdx_check.py --fix` will insert the
header for you.

## NOTICES files are auto-generated

Don't hand-edit `THIRD_PARTY_NOTICES.md` (root) or
`plugins/pencheff/THIRD_PARTY_NOTICES.md`. Both are regenerated by
`tools/generate_notices.py` from the live dependency manifests +
`tools/license-allowlist.txt`. CI fails when the committed file is
stale.

If you've added a new third-party dependency:

1. Add it to `tools/license-allowlist.txt` with its license.
2. Run `python tools/generate_notices.py --write`.
3. Commit both files in the same PR.

## CLA (Contributor License Agreement)

For contributions larger than a typo or one-line fix, the maintainer
may request a signed CLA before merge. The CLA grants Pencheff the
right to relicense your contribution if the project ever needs to
change its outbound license (e.g. dual-license for embedded customers).
The DCO is sufficient for routine contributions.

## What gets a quick "no"

* Adding a competitor's data feed / rule pack / proprietary format
  parser without explicit license documentation.
* Adding a dependency that pulls a GPL transitive.
* Code that downloads model weights or proprietary binaries at runtime
  without a license-acknowledgment opt-in.
* Removing the SPDX header from existing files.
* Hand-edits to `THIRD_PARTY_NOTICES.md`.

## What gets a fast review

* A new SAST / DAST scanner integration following the
  `bench/runners/` shell-runner pattern + a normalizer in
  `apps/api/pencheff_api/services/repo_findings.py`.
* New Pulse templates (`bench/rules/` / community drop-in) with clear
  per-template attribution.
* New OSV / NVD / EPSS / KEV / RustSec / GoVulnDB / distro feed sources
  via the `FeedSource` protocol (see Phase 1 of the IP-clean plan).
* Test coverage for any of the above.

## Reporting a security issue

Don't open a public issue. Email the maintainer (address in the README)
or use the GitHub private-vulnerability-reporting flow on this repo.
